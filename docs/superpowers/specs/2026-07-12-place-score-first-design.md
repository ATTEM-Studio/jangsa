# 플레이스 점수 우선 진단 설계

## 목표

사장님이 긴 체크리스트를 판정하는 화면을 없애고, 플레이스 링크에서 얻은 관찰값을 기준으로 장사네비게이션이 점수와 개선 우선순위를 먼저 설명한다.

## 점수 원칙

- 점수는 매출이나 순위를 보장하지 않는 `손님이 방문 결정을 내리기 쉬운 정도`다.
- 모르는 항목은 0점이 아니라 점수 계산에서 뺀다. 확인 범위가 60%보다 낮으면 총점을 숨긴다.
- 모든 점수에는 한글 근거, 확인 방법, 바로 고칠 수 있는 행동이 있다.
- 자동 수집기가 아직 확인하지 못한 정보는 자동 분석 결과처럼 보이지 않는다. 사장님 확인은 접힌 보완 기능으로 제공한다.

## 100점 점수판

| 영역 | 배점 | 손님 관점의 기준 |
| --- | ---: | --- |
| 대표사진 | 15 | 첫 사진만 보고 음식·서비스와 분위기를 이해할 수 있는가 |
| 상세설명 | 15 | 누구에게 어떤 경험을 주는지 자연스럽게 알 수 있는가 |
| 찾아오는 길 | 10 | 주차·랜드마크·도보 동선이 분명한가 |
| 대표키워드 | 10 | 지역과 대표 메뉴를 함께 찾을 수 있는가 |
| 스마트콜 | 8 | 전화 문의와 안심번호가 바로 쓸 수 있는가 |
| 메뉴정보 | 14 | 사진·설명·가격으로 주문 결정을 도울 수 있는가 |
| 부가정보 | 8 | 주차·화장실·단체·반려동물 등 방문 전 정보가 있는가 |
| 톡톡 | 6 | 자주 묻는 질문과 문의 시작점이 있는가 |
| 예약 | 7 | 방문 목적에 맞게 시간과 인원을 고를 수 있는가 |
| 리뷰 쿠폰 전략 | 7 | 무리한 요구 없이 실제 후기를 받을 장치가 있는가 |

각 영역은 `잘 되어 있음`, `조금 아쉬움`, `비어 있음`, `아직 확인 못 함` 네 상태를 갖는다. 각각 100%, 50%, 0%, 계산 제외로 환산한다.

## 화면 흐름

1. 링크를 입력하면 분석 중 화면에서 사진·설명·메뉴·방문 정보·연락 및 예약·후기 장치를 확인한다고 안내한다.
2. 결과 첫 화면은 `플레이스 점수`, `가장 먼저 고칠 3가지`, `고치면 기대되는 점수`만 보여 준다.
3. 각 영역 카드는 현재 점수와 한 줄 이유만 보여 주고, `왜 이렇게 봤나요?` 토글에서 관찰 근거와 확인 기준을 연다.
4. 자동 분석에 빈칸이 있으면 `직접 확인해서 점수 보완하기` 토글 안에서만 선택지를 제공한다.

## 데이터 계약

`Observation`은 항목 키, 상태, 출처(자동 분석·사장님 확인·계산), 확인 시각, 신뢰도, 근거 문장, 확인 방법을 가진다. 실제 수집기는 이 계약에 맞춰 사진·설명·메뉴·기능 정보를 전달한다. 화면과 점수 엔진은 수집 방식에 의존하지 않는다.

## 오류와 정직성

- 현재 정적 화면은 네이버 플레이스를 실시간으로 읽거나 사진을 인공지능으로 판정하지 않는다.
- 따라서 수집 결과가 없을 때 임의의 점수나 "자동 분석 완료" 문구를 만들지 않는다.
- 샘플 체험은 `예시 매장 분석`임을 분명히 표시하며, 실제 점수와 섞지 않는다.

## 검증

- 총 배점은 100점이다.
- 점수 영역명·근거·선택지는 모두 한글이다.
- 화면 첫 진입에서 영어 판정값과 긴 필수 설문이 보이지 않는다.
- 점수 근거는 접기/펼치기로 확인할 수 있다.
- 미확인 데이터는 0점이 아니며, 범위가 부족하면 총점을 숨긴다.
