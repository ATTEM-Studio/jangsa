# 장사네비게이션 v12 설명 가능한 진단 코어 설계

## 1. 제품 정의

장사네비게이션은 점수를 보여주는 플레이스 검사기가 아니다. 외부에서 보이는 매장 상태와 사장님만 아는 경영 상황을 함께 확인하고, 가장 먼저 실행할 행동 하나를 근거와 함께 정해주는 자영업자용 실행 코치다.

핵심 약속은 다음과 같다.

> 링크 하나로 시작하고, 확인되지 않은 사실은 묻고, 계산 가능한 근거만 사용해 오늘의 행동 하나를 제시한다.

## 2. 이번 설계의 범위

이번 명세는 모든 후속 기능의 기준이 되는 진단 코어와 사용자 경험을 정의한다.

포함 범위:

- 링크 입력과 수동 진단 진입
- 자동 수집 결과의 표준 데이터 형식
- 사장님 확인 및 수정
- 링크로 알 수 없는 경영 질문
- 외부 매장 준비도 점수
- 내부 성장 병목 판단
- 행동 후보 필터링과 우선순위 계산
- 목표점수 계산
- 결과 화면과 CTA
- 실행 결과 기록을 위한 인터페이스
- 오류·미확인·낮은 신뢰도 처리

별도 하위 프로젝트:

1. 네이버 플레이스 실제 수집기와 이용 조건 검토
2. 계정·복수 매장·실행 기록 동기화
3. 카카오 등 외부 리마인드 채널
4. 실사용 데이터 기반 업종 벤치마크

진단 코어는 수집기 종류와 무관하게 같은 입력 계약을 사용한다. 실제 수집기가 준비되지 않았거나 실패해도 사장님 확인 입력으로 진단을 계속할 수 있어야 한다.

## 3. 성공 기준

- 사장님이 링크 입력 후 2분 이내에 첫 행동을 확인한다.
- 결과 첫 화면에서 10초 안에 `무엇을`, `왜`, `어떻게`, `언제 확인할지`를 이해한다.
- 모든 점수는 배점, 출처, 확인 여부를 설명할 수 있다.
- 미확인 항목은 0점으로 계산하지 않는다.
- 실행 불가능하거나 법적 선행 조건이 없는 행동은 추천하지 않는다.
- 목표점수는 실제로 회복 가능한 항목의 배점 합으로만 계산한다.
- 링크 분석 실패 시에도 수동 확인을 통해 진단을 완료할 수 있다.
- 추천 수락, 거절 이유, 실행 결과를 다음 진단에 활용할 수 있는 구조로 저장한다.

## 4. 사용자 여정

### 4.1 링크 입력

첫 화면의 기본 CTA는 `우리 가게 2분 진단 시작`이다. 네이버 플레이스 URL 또는 공유 링크를 받는다. 링크가 없거나 지원되지 않으면 `링크 없이 직접 진단`을 제공한다.

### 4.2 분석 과정 표시

분석 중에는 실제로 확인하는 항목만 단계별로 표시한다.

1. 매장 기본정보
2. 메뉴·사진·편의시설
3. 리뷰·답글·최근성
4. 예약·주문·문의·쿠폰 기능
5. 상세설명·키워드·최근 소식
6. 점수와 개선 후보 계산

확인하지 않은 수치나 실제보다 과장된 건수는 표시하지 않는다. 각 단계는 `완료`, `확인 중`, `확인 불가` 상태를 가진다.

### 4.3 사장님 확인

자동 확인된 정보는 다음 세 상태로 보여준다.

- 자동 확인됨
- 사장님 확인 필요
- 확인할 수 없음

사장님은 잘못된 값을 수정할 수 있다. 자동값과 수정값을 모두 보관하며 최종 점수에는 확인된 최신값을 사용한다.

### 4.4 경영 상황 질문

링크로 알 수 없는 내용만 묻는다.

- 지금 가장 큰 고민
- 이번 달 현재 매출과 목표 매출
- 남은 영업일
- 평균 객단가
- 추가 손님을 받을 수 있는지
- 메뉴·세트 구성을 바꿀 수 있는지
- 온라인 광고 여부와 확인 가능한 광고 수치
- 홍보 수신동의 고객 DB 여부
- 재방문율을 알고 있는지

필수 질문은 추천에 반드시 필요한 최소값으로 제한한다. 모르는 숫자는 `모름`을 선택할 수 있으며, 이 경우 기준 숫자 확인 자체가 오늘의 행동이 될 수 있다.

### 4.5 결과

결과는 다음 순서로 제시한다.

1. 오늘의 결론
2. 오늘의 행동 하나
3. 선택 이유와 사용한 핵심 숫자
4. 3단계 실행법
5. 하지 말아야 할 일
6. 실행 후 확인할 숫자
7. 현재 점수와 목표점수
8. 외부 매장 준비도 세부 점수
9. 내부 성장 병목
10. 출처·미확인·가정 상세
11. 실행 완료 및 결과 기록 CTA

## 5. 데이터 구조

### 5.1 관측값

모든 자동 수집 및 사용자 확인 정보는 같은 구조를 사용한다.

```text
Observation
- key: 항목 식별자
- value: 정규화된 값
- source: collector | owner | calculated
- confidence: high | medium | low
- observedAt: 확인 시각
- status: confirmed | inferred | unknown | unavailable
- evidence: 사용자에게 보여줄 짧은 근거
```

자동값을 사장님이 수정하면 `source=owner`, `status=confirmed`인 새 관측값을 만든다. 원래 자동값은 감사 기록으로 남기되 계산에는 사용하지 않는다.

### 5.2 점수 항목

```text
ScoreItem
- key
- category
- weight
- state: pass | partial | fail | unknown
- earnedPoints
- maxPoints
- observationKeys
- reason
- fixActionKey
```

### 5.3 행동 후보

```text
ActionCandidate
- key
- title
- requiredConditions
- disqualifyingConditions
- impact
- urgency
- feasibility
- costBurden
- confidence
- steps
- metric
- avoid
- recoverableScoreItems
```

## 6. 외부 매장 준비도 점수

외부 매장 준비도만 100점 점수로 표시한다. 매출 성과나 사업 건강도를 링크 정보와 임의로 합산하지 않는다.

| 영역 | 배점 | 주요 항목 |
|---|---:|---|
| 발견 가능성 | 20 | 업종·지역·대표키워드·찾아오는 길·검색 의도 일치 |
| 선택·전환 | 25 | 대표사진·메뉴·가격·예약·주문·문의·쿠폰 |
| 리뷰·신뢰 | 20 | 리뷰 품질·최근성·답글·사진 리뷰·부정 리뷰 대응 |
| 콘텐츠 완성도 | 20 | 상세설명·메뉴설명·소식·사진 다양성·차별점 |
| 운영 활동성 | 15 | 최근 업데이트·예약 슬롯·답글 속도·기능 유지 |

개별 항목 계산:

- 통과: 배점 100%
- 부분 충족: 배점 50%
- 미충족: 배점 0%
- 미확인: 계산 분모에서 제외

카테고리 점수는 확인된 항목의 비율로 환산한다. 전체 배점 기준 확인 범위가 60% 미만이면 종합점수를 표시하지 않고 `정보 추가 필요`를 표시한다. 점수 옆에는 항상 확인 범위를 표시한다.

## 7. 내부 성장 병목

내부 경영 정보는 억지로 하나의 점수로 합치지 않는다. 다음 영역별 상태와 가장 큰 병목을 표시한다.

- 목표 매출 차이와 하루 필요 손님 수
- 객단가 개선 여지
- 광고 안전 기준과 클릭 허용 범위
- 재방문 구조
- 추가 수요를 받을 운영 여력
- 실행 가능한 채널과 보유 자산

병목은 `현재 문제`, `확인된 근거`, `아직 필요한 숫자`로 설명한다.

## 8. 행동 우선순위 엔진

추천 후보는 먼저 선행 조건을 검사한다.

- 수신동의가 없으면 고객 메시지를 추천하지 않는다.
- 추가 손님을 받을 수 없으면 신규 유입 확대를 추천하지 않는다.
- 메뉴 변경이 불가능하면 메뉴·세트 행동을 추천하지 않는다.
- 필수 기준 숫자가 없으면 지출 행동보다 기준 숫자 확인을 우선한다.
- 목표를 이미 달성했다면 무리한 유입보다 이익·재방문 방어를 우선한다.

조건을 통과한 후보에 다음 점수를 계산한다.

```text
우선순위 점수 =
예상 영향 35%
+ 긴급성 20%
+ 실행 가능성 20%
+ 비용 부담이 낮음 10%
+ 근거 신뢰도 15%
```

점수는 엔진 검증용이며 사용자에게 절대적인 성공 확률처럼 표시하지 않는다. 사용자는 `왜 이 행동이 선택됐는지`와 주요 근거를 본다.

## 9. 목표점수

목표점수는 추천된 행동으로 회복 가능한 외부 점수 항목의 배점만 더한다.

```text
목표점수 = 현재점수 + 선택된 개선 항목의 미획득 점수
```

예시:

```text
현재 74점
+ 예약 기능 3점
+ 대표사진 4점
+ 최근 소식 2점
= 1차 목표 83점
```

미확인 항목, 내부 매출 추정, 업종 평균은 목표점수에 임의로 더하지 않는다.

## 10. 업종 비교

실제 비교 데이터가 없을 때는 `업종 평균`이라는 표현을 사용하지 않는다. 초기 버전에서는 전문가 기준표와 비교하며 `권장 기준 대비`라고 표시한다.

실사용 벤치마크는 다음 조건을 모두 만족할 때만 공개한다.

- 같은 업종의 유효 표본 30개 이상
- 동일한 점수 버전 사용
- 최근 90일 이내 데이터
- 개인정보 제거
- 극단값 처리 규칙 적용

조건을 충족하지 못하면 더 넓은 업종군으로 비교하거나 비교 자체를 숨긴다. 표본 수와 기준일을 함께 표시한다.

## 11. CTA와 무료·가입 경계

로그인 없이 제공:

- 종합 상태 또는 정보 추가 필요 상태
- 영역별 점수
- 권장 기준 비교
- 오늘의 행동 하나
- 행동 선택 이유
- 3단계 실행법
- 목표점수

가입 후 제공:

- 전체 30일 처방 순서
- 진단과 수정 이력 저장
- 3일·7일 결과 기록
- 전후 점수 및 매출지표 비교
- 다음 행동 자동 추천
- 리마인드

가입 CTA는 결과를 인질로 잡지 않는다. `전체 결과 보기`가 아니라 `이 진단을 저장하고 다음 행동 이어가기`로 표현한다.

행동 카드 CTA 우선순위:

1. `지금 고치기` 또는 실제 관리 화면 이동
2. `AI로 초안 만들기`
3. `따라 하기 가이드`
4. `오늘은 어려워요`와 대체 행동

## 12. 시스템 경계

진단 코어는 다음 모듈로 분리한다.

- `collector-adapter`: 외부 정보를 Observation으로 변환
- `observation-store`: 자동값과 사용자 수정값 관리
- `score-engine`: 외부 준비도 계산
- `business-metrics`: 매출·광고·객단가·재방문 계산
- `action-engine`: 조건 필터와 우선순위 선정
- `diagnosis-presenter`: 사용자용 근거와 결과 모델 생성
- `progress-store`: 수락·거절·완료·결과 기록
- `ui-flow`: 링크 입력부터 결과까지 화면 상태 관리

수집 방식이 바뀌어도 점수 엔진과 UI 계약은 바뀌지 않아야 한다.

## 13. 오류 처리

### 링크 오류

- 잘못된 URL: 올바른 공유 링크 예시 제공
- 지원되지 않는 URL: 직접 진단으로 전환
- 매장 비공개·삭제: 상태 설명 후 수동 입력 제공
- 수집 시간 초과: 확인된 항목만 유지하고 나머지는 사장님 확인으로 전환

### 데이터 오류

- 자동값과 사장님 답변이 다르면 사장님 확인값 우선
- 오래된 정보는 날짜를 표시하고 재확인 요청
- 필수 관측값이 부족하면 점수를 숨기고 부족한 정보만 요청
- 계산 불가능한 효과는 범위를 만들지 않고 `실행 후 측정`으로 표시

### 추천 오류 방지

- 모든 행동은 필수 조건과 제외 조건을 가진다.
- 낮은 신뢰도 추천은 큰 지출을 요구할 수 없다.
- 광고·메시지·개인정보 관련 행동은 안전 조건을 먼저 검사한다.
- 매출 보장, 순위 보장, 과장된 백분위 표현을 금지한다.

## 14. 개인정보와 신뢰

- 공개된 매장 정보와 사장님 입력을 구분한다.
- 민감한 원본 매출·광고 수치를 공유 URL에 넣지 않는다.
- 가입 전 데이터는 브라우저 임시 저장을 기본으로 한다.
- 서버 저장 시 보관 목적과 삭제 방법을 안내한다.
- 진단 결과에는 점수 버전과 진단 시각을 표시한다.
- 공유 카드는 민감한 매출 수치 없이 점수·유형·행동만 포함한다.

## 15. 테스트 전략

### 단위 테스트

- pass·partial·fail·unknown 배점
- 미확인 분모 제외
- 60% 확인 범위 기준
- 목표점수 합산과 100점 상한
- 각 행동의 필수·제외 조건
- 우선순위 동점 규칙
- 기존 매출·광고·객단가 계산 회귀 테스트

### 균형 테스트

업종·고민·수용 여력·DB·광고·메뉴 변경 가능성의 조합을 포함한 균형 시나리오를 유지한다. 특정 업종이 무조건 같은 행동을 받지 않는지 확인한다.

### 통합 테스트

- 링크 성공 → 확인 → 질문 → 결과
- 일부 수집 실패 → 수동 확인 → 결과
- 링크 실패 → 직접 진단 → 결과
- 미확인 60% 미만 → 점수 숨김
- 행동 수락 → 기록 → 7일 결과 입력
- 행동 거절 → 대체 행동 제시

### 사용자 검증

외식업 사장님 5명 이상에게 다음을 확인한다.

- 결과 첫 화면에서 해야 할 일을 10초 안에 말할 수 있는가
- 점수의 이유를 이해하는가
- 모르는 항목이 0점 처리되지 않는다는 사실을 이해하는가
- 목표점수가 현실적으로 느껴지는가
- CTA가 강요가 아니라 도움으로 느껴지는가
- 실제로 행동을 완료했는가

## 16. 구현 순서

1. Observation·ScoreItem·ActionCandidate 데이터 계약
2. 설명 가능한 점수 엔진
3. 기존 행동 엔진에 신뢰도와 선행 조건 강화
4. 자동 확인·사장님 수정 UI
5. 외부 준비도와 내부 병목 결과 UI
6. 목표점수와 행동 CTA
7. 실패 시 직접 진단 전환
8. 실행 기록과 재진단 연결
9. 실제 수집기 연동
10. 충분한 표본 확보 후 업종 벤치마크

## 17. 완료 기준

- 같은 입력은 항상 같은 점수와 행동을 만든다.
- 모든 결과 항목에서 근거와 출처를 추적할 수 있다.
- 자동 수집이 전부 실패해도 직접 진단을 완료할 수 있다.
- 미확인 데이터 때문에 부당하게 점수가 낮아지지 않는다.
- 실행 불가능한 행동이 추천되지 않는다.
- 결과 첫 화면에 행동·이유·3단계·하지 말 것·확인 숫자·목표점수가 모두 존재한다.
- 전체 자동 검증과 실제 사용자 검증 기준을 통과한다.
